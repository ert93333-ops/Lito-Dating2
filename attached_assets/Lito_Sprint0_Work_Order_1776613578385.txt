# Lito Sprint 0 작업지시서

작성일: 2026-04-19
목적: Closed Beta 이전, 런칭 차단 P0 이슈 제거
기준 자료: 코드베이스 감사 결과

---

## 1. 이번 스프린트 목표

이번 스프린트의 목표는 **새 기능 추가가 아니라 런칭 차단 이슈 제거**다.

이번 배치가 끝나면 아래가 가능해야 한다.

1. 비인증 사용자가 `/api/ai/*`를 호출할 수 없다.
2. `SESSION_SECRET` 없이 서버가 부팅되지 않는다.
3. 차단한 사용자가 discover에 노출되지 않는다.
4. Apple Sign-In이 안전하게 검증되거나 배포 대상에서 제거된다.
5. EAS 빌드가 실제로 실행 가능하다.

---

## 2. 현재 블로커 요약

감사 결과 기준 현재 P0 블로커는 아래 5개다.

1. `/api/ai/*` 전체에 `requireAuth` 없음
2. `SESSION_SECRET` 폴백 시크릿 사용
3. 차단 유저가 discover에 계속 노출
4. `eas.json` 없음
5. Apple Sign-In JWKS 검증 없음

## 6. 세부 작업지시

## Task 1. `/api/ai/*` 전체 인증 강제

### 목적
비인증 사용자의 OpenAI API 무단 사용을 차단한다.

### 배경
감사 결과 `/api/ai/suggest-reply`, `/api/ai/translate`, `/api/ai/coach` 등 AI 라우트에 `requireAuth`가 누락되어 있다.

### 요구사항
- 모든 production-facing `/api/ai/*` 엔드포인트에 `requireAuth` 적용
- 이미 rate limit이 있다면 `requireAuth`보다 앞/뒤 순서도 점검
- 테스트/내부용 라우트가 있다면 명시적으로 분리
- 비로그인 접근 시 401 또는 403 일관 반환
- 클라이언트가 인증 누락 상태에서 호출하는 기존 코드가 있다면 해당 호출부도 정리

### 확인 대상
- `routes` 또는 `modules/interest`, `modules/coaching`, `translate`, `suggest-reply` 관련 라우트
- API router mounting 지점

### 산출물
- 수정 파일 목록
- 인증 적용된 엔드포인트 목록
- 비인증 접근 테스트 결과

### 완료 기준
- 비로그인 상태에서 `/api/ai/*` 호출 시 OpenAI 호출까지 가지 않고 차단됨
- 로그인 상태에서는 기존 기능 정상 작동

---

## Task 2. `SESSION_SECRET` 필수 환경변수 강제화

### 목적
개발용 폴백 시크릿으로 인한 JWT 위조 가능성을 제거한다.

### 배경
감사 결과 `lib/jwt.ts`에서 폴백 시크릿이 남아 있다.

### 요구사항
- 프로덕션/비개발 환경에서 `SESSION_SECRET` 미설정 시 서버 시작 실패
- 개발 환경에서만 명시적인 dev fallback 허용 여부를 결정하고 코드상 분리
- `.env` 템플릿, 배포 문서, Replit/Expo 환경변수 가이드 업데이트
- 에러 메시지는 명확해야 함

### 확인 대상
- `lib/jwt.ts`
- 서버 부트스트랩 파일
- env 문서 및 템플릿

### 산출물
- 환경변수 체크 로직
- 실행 실패 스크린샷 또는 로그
- 필요한 env 목록 업데이트

### 완료 기준
- `SESSION_SECRET` 없이 production mode 부팅 불가
- 정상 값 설정 시 로그인/JWT 검증 정상 동작

---

## Task 3. 차단 유저 discover 필터링

### 목적
차단한 사용자가 추천/디스커버 목록에 노출되지 않게 한다.

### 배경
감사 결과 `user_blocks`는 있으나 discover 쿼리에 반영되지 않는다.

### 요구사항
- discover query에서 아래 모두 제외
  - 내가 차단한 사용자
  - 나를 차단한 사용자
- mock/demo/real user 혼합 구조를 깨지 않게 처리
- 차단 상태가 즉시 반영되도록 확인
- 향후 match/chat에도 동일 원칙을 적용할 여지가 있는지 TODO 명시

### 확인 대상
- `modules/match/match.service.ts`
- `user_blocks` 관련 repository/query
- discover response shaping 코드

### 산출물
- 수정된 discover 쿼리
- 차단/역차단 케이스 테스트 결과

### 완료 기준
- 차단 후 discover 새로고침 시 해당 사용자가 노출되지 않음
- 역차단도 노출되지 않음

---

## Task 4. Apple Sign-In 보안 보강 또는 임시 비활성화

### 목적
임의 Apple UID로 계정 생성/탈취 가능한 취약점을 제거한다.

### 배경
감사 결과 Apple Sign-In은 클라이언트가 넘긴 `providerUserId`를 신뢰하고 있고, JWKS 기반 ID token 검증이 없다.

### 요구사항
두 가지 경로 중 하나를 선택하되, 먼저 현재 코드 상태를 보고 더 안전한 쪽을 채택한다.

#### 옵션 A — 올바른 수정
- Apple identity token 검증 구현
- Apple JWKS 조회 및 서명 검증
- iss, aud, exp, sub 검증
- 검증된 sub만 providerUserId로 사용

#### 옵션 B — 임시 비활성화
- Apple 로그인 버튼/엔드포인트를 production path에서 제거 또는 feature flag로 차단
- Google/Kakao/Line/email만 유지
- 문서에 Apple 임시 비활성화 사유 명시

### 산출물
- 선택한 옵션
- 이유
- 영향 파일 목록
- 보안 검증 방식 설명

### 완료 기준
- 더 이상 client-supplied Apple UID만으로 계정 생성 불가

---

## Task 5. EAS 빌드 및 배포 경로 복구

### 목적
앱스토어/테스트플라이트/안드로이드 배포용 실제 빌드 경로를 확보한다.

### 배경
감사 결과 `eas.json`이 없고 Expo project ID도 누락되어 있다.

### 요구사항
- `eas.json` 추가
- app.json/app.config의 Expo project ID 설정
- 필요한 빌드 프로필 정의
  - development
  - preview
  - production
- iOS/Android identifier와 충돌 없는지 재확인
- 최소 1회 빌드 커맨드 기준 정리
- 필수 env 목록 문서화

### 산출물
- `eas.json`
- Expo project config 수정
- 빌드 실행 방법 문서

### 완료 기준
- EAS build 명령 기준으로 실제 빌드 가능한 상태
- 누락 env가 문서화되어 있음
