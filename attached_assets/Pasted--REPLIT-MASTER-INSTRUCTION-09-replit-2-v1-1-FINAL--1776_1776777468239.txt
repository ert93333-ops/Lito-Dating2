==================================================
0. 현재 코드베이스 판정
==================================================

현재 리포는 아래 상태다.

1. growth / monetization / profile coach / referral / analytics는 이미 client-side mock foundation이 있음
2. upgradePlan은 mock upgrade이며 실제 store receipt 검증 구조가 아님
3. analytics는 facade만 있고 실제 canonical server event 저장 구조가 아님
4. admin은 mockData 기반 대시보드이며 실제 read API 연결이 아님
5. report / safety는 client-side 흐름과 타입이 먼저 있고, 서버 저장 및 moderation queue 연동은 미완료
6. 이번 스프린트는 UI 추가보다 server-authoritative patch가 핵심임

따라서 이번 작업은
“새 화면 만들기”가 아니라
“기존 mock 구조를 실제 서버/검증/이벤트 구조로 교체”
하는 것이다.

3. 현재 리포 기준 핵심 판단
==================================================

현재 리포에는 GrowthContext, monetization, analytics facade, profile coach UI/state, admin mock dashboard가 이미 있으므로
이번 구현은 아래처럼 진행해야 한다.

- GrowthContext는 “서버 응답 소비자”로 축소
- billing / wallet / ledger / coach / analytics / translation 상태 판정은 서버가 담당
- admin은 mockData 렌더링 구조를 최대한 유지하되 API fetch로 교체
- 채팅 translation 쪽은 기존 UI/렌더링을 깨지 말고 backend event만 실제화
- safety/reporting은 기존 타입을 서버 persistence와 queue 조회로 연결

--------------------------------------------------
A. Analytics canonical event 서버화
--------------------------------------------------

목표:
현재 client facade analytics를 서버 canonical event 기준으로 전환한다.

해야 할 일:
1. 서버에 canonical analytics ingest 레이어 추가
2. 아래 이벤트를 enum/constant로 서버 기준 잠금
3. 클라이언트는 track 호출 시 서버 event endpoint 또는 서버 action을 통해 기록
4. 서버는 최소한 event_name, actor_id, target_id(optional), conversation_id(optional), props(json), created_at 저장
5. raw prompt/response 저장 금지
6. dev console-only analytics 남아 있으면 production path에서 제거
7. GrowthContext / paywall / profile coach / report / delete / chat translation 관련 호출을 서버 canonical event로 연결

필수 이벤트:
- auth_completed
- age_gate_passed
- required_consents_completed
- profile_translation_preview_requested
- profile_published
- candidate_viewed
- match_created
- first_message_sent
- first_reply_received
- translation_requested
- message_translation_rendered
- translation_retry
- original_text_viewed
- coach_opened
- coach_request_started
- coach_request_completed
- coach_blocked_unsafe
- coach_blocked_zero_credit
- coach_blocked_no_consent
- profile_coach_saved
- report_submitted
- block_user_completed
- delete_request_started_in_app
- delete_web_flow_opened
- delete_request_submitted
- delete_completed

주의:
analytics.ts 같은 기존 facade는 삭제하지 말고 server transport wrapper로 바꿔라.

--------------------------------------------------
B. Translation provider 실제 연동
--------------------------------------------------

목표:
기존 placeholder translation path를 실제 provider adapter 구조로 연결한다.

해야 할 일:
1. callTranslationProvider 또는 동등 구현 지점 찾기
2. provider adapter 분리
3. timeout / retry / fallback 처리
4. consent 없으면 provider 호출 금지
5. message insert와 translation success를 완전히 분리
6. status = pending / success / failed / consent_required 저장
7. translation 실패 시 메시지 전송 실패로 간주하지 말 것
8. 원문 조회 original_text_viewed 흐름 유지
9. retry 이벤트와 original view 이벤트 analytics 연동

필수 보장:
- message는 살아있고 translation만 실패 가능해야 함
- 번역은 무료
- translation failure가 chat send failure가 되면 안 됨

--------------------------------------------------
C. Profile coach 서버 구현
--------------------------------------------------

목표:
현재 client-side suggestion 흐름을 서버 authoritative coach API로 교체한다.

구현 엔드포인트:
- POST /v1/ai/profile-coach
- POST /v1/ai/profile-coach/{coach_output_id}/save

입력:
- profile snapshot
- locale / target audience context
- consent status
- user id

서버 규칙:
1. consent 없으면 외부 AI 호출 금지
2. unsafe_interaction이면 blocked response 반환
3. zero_credit이면 blocked response 반환
4. validation 실패면 structured error 반환
5. 성공 반환 시에만 ai_ledger debit
6. profile에 자동 overwrite 금지
7. save 호출 시 선택 반영만 수행
8. raw prompt/response 장기 저장 금지
9. 저장 시 최종 suggestion/result metadata만 최소 저장

반환 형식:
- success: suggestions[], coach_output_id, remaining_credit(optional)
- blocked_no_consent
- blocked_zero_credit
- blocked_unsafe
- validation_error

클라이언트 작업:
- GrowthContext의 refreshProfileSuggestions / accept / reject 흐름을 서버 응답 소비 구조로 수정
- 기존 UI는 최대한 유지

--------------------------------------------------

목표:

필수 상태 흐름:
1. delete_request_started_in_app
2. delete_web_flow_opened
3. delete_request_submitted
4. visibility hidden

7. revoke_provider
8. purge_profile
9. purge_messages
10. purge_translations

해야 할 일:

--------------------------------------------------
F. Admin read API 구현
--------------------------------------------------

목표:
현재 mockData 기반 admin dashboard를 실제 read API 소비 구조로 전환한다.

최소 범위:
- report queue 조회
- user detail 조회
- delete request list 조회
- delete request detail 조회
- wallet/purchase summary 조회
- risk flags 조회
- moderation actions 조회

구현 규칙:
1. 기존 admin UI 구조는 최대한 유지
2. mockData.ts는 제거하지 말고 fallback/dev seed 용도로 남겨도 됨
3. production path는 API fetch 기준이어야 함
4. raw prompt 장기 노출 금지
5. 신고 증거는 최소 열람 원칙
6. audit log 필수

--------------------------------------------------
G. WebSocket 이벤트 완성
--------------------------------------------------

목표:
실시간 translation / safety 상태 변화를 conversation 화면에 반영한다.

필수 이벤트:
- chat.translation_updated
- chat.safety_state_updated

해야 할 일:
1. translation status update 시 websocket publish
2. safety state change 시 websocket publish
3. reconnect/backfill 시 translation status 복원
4. conversation lock 이후도 실시간 반영
5. chat UI가 기존 translation cache/render 흐름을 유지하도록 최소 침습 patch

주의:
frontend에서 translation state를 임시 낙관적으로 fake 완료 처리하지 말 것.
서버 authoritative 상태를 기준으로 갱신하라.
