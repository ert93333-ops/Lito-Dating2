==================================================
0. 현재 코드베이스 판정
==================================================

이 작업은 새 프로젝트 생성이 아니다.
반드시 현재 같은 Lito-Dating2 리포를 이어서 패치한다.

현재 모바일 앱에는 이미 아래가 존재한다.

1. GrowthContext
- subscription
- AI coach credit
- profileSuggestions
- referral
- analytics pass-through
- mockUpgradeToPlan / mockAddConsumable 계열 구조 포함

2. 화면/라우트
- paywall screen
- profile-coach screen
- profile screen 내 membership & AI section
- matches/discover 쪽 growth entry
- GrowthProvider 연결 구조

3. 현재 상태의 문제
- 결제는 mock upgrade 구조가 남아 있음
- profile coach는 local suggestion 생성 기반임
- analytics는 facade 중심임
- coach UX가 실제 서버 blocked state(no_consent / zero_credit / unsafe) 기준으로 잠기지 않았을 가능성이 큼
- purchase success 이후 원래 draft/context 복구 같은 상용 UX가 아직 명확히 연결되지 않았음

따라서 이번 작업의 핵심은
“새 백엔드를 만드는 것”이 아니라
“이미 있는 모바일 growth UI를 서버 authoritative 기능과 정확히 연결하는 것”이다.

1. 절대 목표
==================================================

아래 6개를 모바일 앱에서 실제 동작하도록 연결하라.

1. chat room의 conversation coach 진입 UX
2. zero_credit → paywall → purchase success return UX
3. unsafe_interaction → coach 차단 + safety CTA UX
4. no_consent → consent sheet UX
5. profile editor의 profile coach preview/apply UX
6. analytics / wallet / purchase / block reason UI 반영

==================================================
2. 절대 원칙
==================================================

1. 번역은 코어다. 코칭보다 앞선다.
2. 기본 채팅과 기본 번역은 무료다.
3. 코칭은 선택형 보조 기능이다.
4. 코칭 자동 실행 금지.
5. 사용자의 명시적 요청 이후에만 coach API 호출.
6. zero_credit이어도 채팅과 번역은 계속 가능해야 한다.
7. unsafe_interaction이면 paywall보다 신고/차단/안내가 우선이다.
8. no_consent 상태에서는 결제보다 동의가 먼저다.
9. 결과는 자동 전송 금지. 항상 사용자가 선택/수정/전송해야 한다.
10. 현재 mock client logic가 남아 있다면 지우지 말고 서버 authoritative 응답 consumer로 교체한다.
11. 기존 Expo UI를 불필요하게 갈아엎지 말고 patch 방식으로 수정한다.
12. 파일명/경로를 추정하지 말고 실제 리포 기준으로 수정한다.

3. 현재 리포 기준 핵심 판단
==================================================

이번 작업에서 반드시 이렇게 판단하라.

1. GrowthContext는 계속 유지하되 역할을 바꾼다.
- 기존: mock state owner
- 변경: server state consumer + UI convenience wrapper

2. paywall.tsx는 새로 만들지 말고 실제 purchase/verify 결과를 consume하게 바꾼다.

3. profile-coach.tsx는 local suggestion generator를 직접 부르지 말고
기존 profile coach API 결과를 렌더링하는 preview/apply UI로 바꾼다.

4. chat room 내 coach UX는 “대화 대리 작성”이 아니라
사용자가 쓴 draft를 다듬는 polishing tool로만 동작시킨다.

5. unsafe / no_consent / zero_credit은 하나의 공통 block modal로 합치지 말고
각 상태별로 분리된 UI 의미를 유지한다.

--------------------------------------------------
A. Chat Room: Conversation Coach Entry 연결
--------------------------------------------------

목표:
사용자가 채팅방에서 draft를 쓴 뒤에만 AI 코치를 명시적으로 열 수 있게 한다.

반드시 구현:
1. Empty chat state에는 AI가 먼저 문장을 생성하지 않는다.
2. Empty chat state에는 “첫 메시지 도움” chip만 둘 수 있다.
3. chip 탭은 composer에 seed text를 넣을 수 있지만 coach_opened로 집계되면 안 된다.
4. draft 길이가 최소 기준 이상일 때만 coach CTA 활성화.
5. coach CTA 라벨은 아래 의미 안에서만 사용:
   - 다듬기
   - 부드럽게
   - 더 짧게
6. coach CTA 탭 시 상태 분기:
   - unsafe이면 unsafe flow
   - no consent이면 consent flow
   - zero credit이면 zero credit flow
   - 아니면 conversation coach request
7. coach 결과는 bottom sheet 또는 overlay preview로 노출
8. 결과 선택 후 composer에만 반영
9. 자동 전송 금지
10. 적용 후 사용자가 직접 수정 가능해야 한다

핵심 의미:
- 코칭은 대화 대리인이 아니다
- draft polishing tool이다

--------------------------------------------------
B. Conversation Coach 상태 머신 구현
--------------------------------------------------

coach CTA 탭 시 반드시 아래 순서를 따른다.

1. draft validation
2. latest unsafe state 확인
3. consent 확인
4. wallet / trial / credit 확인
5. 서버 coach request 실행
6. result preview 노출
7. 사용자가 선택 후 draft 적용
8. 사용자가 직접 전송

절대 금지:
- coach button 탭 직후 자동 결제
- coach button 탭 직후 자동 전송
- 결과 수신 후 자동으로 composer overwrite
- blocked 상태인데 coach 요청 강행

--------------------------------------------------
C. No Consent Flow 연결
--------------------------------------------------

목표:
AI 기능별 first-use opt-in을 모바일 UX로 정확히 반영한다.

반드시 구현:
1. translation consent와 coach consent를 구분
2. conversation coach와 profile coach consent도 분리 가능 구조 유지
3. no_consent 상태에서는 paywall보다 consent sheet가 먼저
4. consent sheet CTA:
   - 동의하고 계속
   - 지금은 안 함
5. 동의 거절 후에도 기본 채팅/기본 번역은 계속 가능
6. consent 완료 후 원래 context 복구
7. draft / selected mode / active conversation context가 사라지면 안 된다
8. consent 성공 후 coach 자동 재실행 금지
9. 사용자가 다시 CTA를 눌러야 한다

금지:
- consent와 결제를 한 장에서 동시에 요구하는 것
- consent 거절 시 draft 초기화
- consent 성공 후 결과 자동 생성

--------------------------------------------------
D. Zero Credit → Paywall → Return Flow
--------------------------------------------------

목표:
잔액 부족 상태에서도 사용자가 기본 채팅을 계속할 수 있게 하면서,
필요한 경우에만 코칭 pack을 구매하도록 한다.

반드시 구현:
1. zero_credit은 coach 요청 이후에만 평가/노출
2. zero credit sheet에는 아래 의미가 반드시 포함되어야 한다:
   - 기본 채팅과 기본 번역은 계속 무료입니다
3. zero credit sheet CTA:
   - 코칭 팩 보기
   - 그냥 채팅 계속하기
4. paywall은 explicit coach request 이후에만 연다
5. paywall 상품은 3개:
   - Starter
   - Standard
   - Value
6. 기본 강조 pack은 Standard
7. 첫 사용자는 trial이 남아 있으면 trial 먼저 사용
8. trial 규칙:
   - 3회 shared trial
   - success 반환 시에만 소진
   - no expiry
9. purchase success 후:
   - wallet refresh
   - purchase verification state 반영
   - 원래 draft/context 복구
   - coach 자동 재실행 금지
   - 사용자가 다시 코칭 실행하기를 눌러야 함
10. purchase failed / cancelled 후:
   - draft 유지
   - 그냥 채팅 계속 가능
11. no-charge failure는 paywall로 보내지 말 것
12. “차감되지 않았어요” 메시지를 분명히 보여줄 것

금지:
- match 직후 paywall
- 번역 버튼 근처 paywall
- 신고/차단/삭제 근처 paywall
- guilt/urgency/guarantee copy

--------------------------------------------------
E. Unsafe Interaction UX 연결
--------------------------------------------------

목표:
위험 상황에서 코칭을 차단하고 안전 surface를 최우선으로 올린다.

반드시 구현:
1. unsafe_interaction이 true이면 coach CTA 숨김 또는 비활성화
2. safety banner/card 노출
3. primary action:
   - 신고하기
4. secondary action:
   - 차단하기
5. tertiary action:
   - 도움말 보기
6. 번역 / 원문 보기 기능은 유지
7. paywall 금지
8. unsafe category별 표시 가능 구조 유지:
   - money_request
   - investment_crypto
   - off_platform_push
   - sexual_coercion
   - harassment
   - self_harm_risk
   - minor_risk
   - identity_fraud_risk
9. websocket 또는 서버 상태 갱신 시 coach CTA를 즉시 내릴 수 있어야 한다

금지:
- unsafe 상황에서 답장 다듬기 계속 노출
- unsafe 상황에서 결제 제안
- unsafe / zero_credit / no_consent을 하나의 일반 에러 모달로 합치는 것

--------------------------------------------------
F. Profile Coach UX 연결
--------------------------------------------------

목표:
프로필 편집에서 profile coach를 “허위 포장”이 아니라 “표현 정리” 도구로 구현한다.

반드시 구현:
1. profile coach 허용 필드:
   - 자기소개
   - 관심사 설명
   - 관계 의도 설명
   - 문화/언어 자기설명
2. fact field에는 coach 금지:
   - 이름
   - 나이
   - 위치
   - 직업/학교 같은 사실 필드
3. 사용자가 텍스트를 쓴 뒤에만 coach entry 활성
4. profile coach 결과는 preview로 보여준다
5. 원문과 AI 제안을 비교할 수 있어야 한다
6. 자동 overwrite 금지
7. 선택해서 적용
8. 적용 후 직접 수정 가능
9. profile coach 적용 후 translation preview를 다시 거치게 할 것
10. zero_credit / no_consent는 별도 처리
11. high-risk 계정 제한 상태면 coach 차단 가능

현재 리포 반영 규칙:
- existing profile-coach screen을 버리지 말 것
- accept/reject UI는 최대한 유지
- suggestion source만 local generator에서 server response로 교체
- updateProfile 직접 반영은 “사용자 선택 이후”에만 허용

--------------------------------------------------
G. Wallet / Balance UX 연결
--------------------------------------------------

목표:
잔액 정보는 보이되 과금 압박처럼 보이지 않게 한다.

반드시 구현:
1. wallet balance 조회 연결
2. coach 전용 잔액/remaining count 표시 연결
3. low balance는 작은 indicator 정도만
4. low balance만으로 전면 paywall 금지
5. wallet/settings에서 voluntary top-up 허용
6. 채팅방에서는 zero_credit에서만 paywall
7. 잔액 표시는 서버 authoritative 값 사용
8. trial 남은 횟수와 paid credit은 구분 가능하게 할 것
