==================================================
NON-NEGOTIABLE ARCHITECTURE RULES
==================================================

1. Separate interest module
The system must live in a separate backend module, for example:
backend/interest/ or the equivalent existing server module pattern.

2. Directional scoring is mandatory
Scoring must be viewer-relative.
The same conversation must produce different directional outputs depending on the viewer.

Example:
- viewer = A → estimate partner B’s receptivity toward A → PRS_AB
- viewer = B → estimate partner A’s receptivity toward B → PRS_BA

This is mandatory.
Conversation-level undirected scoring is NOT acceptable.

3. Async execution only
Chat message send/store flow must complete normally first.
Only after the message is stored should an async job be triggered.

4. Layered scoring
Layer 1:
- deterministic
- rule-based
- no LLM dependency
- must always run

Layer 2:
- optional LLM semantic enrichment
- enriches feature extraction only
- if it fails, Layer 1 result must still be valid and stored

5. Snapshot storage model
Do NOT store latest snapshot only.
You must support:
- append-only historical snapshots
- fast-read latest snapshot table or equivalent materialized latest state

6. Snapshot deduplication
Do NOT write a new historical snapshot if there is no meaningful change.

7. Incremental feature extraction
Do NOT recompute the full feature window from scratch every time if avoidable.
Use incremental processing where possible.

8. No chat path contamination
Do NOT put PRS/CS computation inside chat service or request handler.
Do NOT call LLM directly from chat endpoints.
Do NOT mutate chat messages for scoring.

CRITICAL HIDDEN FAILURE MODES YOU MUST AUDIT EXPLICITLY
==================================================

You MUST explicitly check and report on each of the following.
Do not skip any item.

A. Directional privacy leak
If PRS is directional, it must NOT be broadcast room-wide as the same payload to all participants.
You must explicitly design viewer-scoped delivery.

B. Auth bypass risk
Do NOT trust viewerUserId from client query/body if the backend already has authenticated user identity.
Viewer identity must be derived from server auth/session except for clearly isolated internal test/admin tooling.

C. Participant source correctness
Do NOT rely only on distinct message senders if a conversation membership source exists.
Use the authoritative participant source if available.
If only distinct senders exist today, state the limitation clearly.

D. Score unit mismatch
PRS and CS must have consistent units across:
- internal scoring
- database
- API
- UI
If internal normalized values use 0..1, boundary conversions must be explicit.
Final stored and served values should be consistently defined.

E. LLM hard dependency risk
If LLM fails, rate-limits, times out, or returns malformed output, scoring must still work.

F. Duplicate async execution / race conditions
Multiple messages or repeated schedule calls must not corrupt or duplicate results.
Design idempotency and per-conversation processing safety.

G. Multi-instance deployment risk
If debounce/locks are in-memory only, this is not safe for multi-instance deployments.
You must explicitly state:
- MVP single-instance assumption, OR
- distributed lock / queue plan (Redis / DB / equivalent)

H. Snapshot explosion risk
Historical snapshots must not grow uncontrollably.
Add deduplication policy with explicit thresholds.

I. Full recomputation cost risk
Repeatedly recomputing from scratch for every update is not acceptable without justification.

J. Wrong viewer delivery
WS updates and read APIs must only return the snapshot belonging to the authenticated viewer direction.

K. PII/privacy over-storage
Do NOT store raw private message text in snapshot tables unless absolutely necessary.
Prefer structured features, reason codes, hashes, and metadata.

L. Double-update noise
If Layer 1 stores a base snapshot and Layer 2 stores an enriched snapshot seconds later, the client experience must not become noisy.
Define when the second update is suppressed or emitted.

M. Message edit/delete/order issues
Explicitly state how the system behaves if messages are edited, deleted, or arrive out of order.

N. Migration safety
Schema changes must be additive and rollback-conscious.
Do NOT propose destructive migrations unless absolutely necessary.
