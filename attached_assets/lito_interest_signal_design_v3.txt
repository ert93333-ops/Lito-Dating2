3. 아키텍처 제안
----------------------------------------------------------------

[데이터 흐름]

WS "message" 수신
  -> chatService.sendMessage(...)     [메시지 저장]
  -> broadcast(room, message)         [채팅 흐름 완료, 클라이언트 응답]
  -> setImmediate(() => {             [논블로킹, 이벤트 루프 다음 틱]
       interestWorker.schedule(
         conversationId,
         savedMessage.id,
         savedMessage.senderUserId
       )
     })

[백그라운드 - 채팅 경로와 완전 분리]

interestWorker
  -> 안전장치 체크 (debounce / lastSnapshot 연령 / 동시 실행 가드)
  -> chatRepository.getMessages(conversationId, limit=50)
  -> participants = distinct senderUserId

  for each viewerUserId in participants:
    analyseOneDirection(conversationId, viewerUserId, messages)

analyseOneDirection:
  1. featureExtractor(messages, viewerUserId)           -> featureWindow
  2. [Layer 1] prsScoring(featureWindow, {0.5,0.5,0.5}) -> baseSnapshot
  3. deduplicationCheck(baseSnapshot, lastSnapshot)
     -> 변화 없으면 SKIP
  4. interestRepo.insertSnapshot(baseSnapshot, llmEnriched=false)
     interestRepo.upsertLatest(baseSnapshot, llmEnriched=false)
  5. wsBroadcastToViewer(viewerUserId, "prs_update", baseSnapshot)
  6. [Layer 2] if circuitBreaker.isOpen -> SKIP
     withRetry(3, backoff)(llmExtract(messages, viewerLang))
       -> semanticScores
     if success AND materialChange(enriched, base):
       interestRepo.insertSnapshot(enrichedSnapshot, llmEnriched=true)
       interestRepo.upsertLatest(enrichedSnapshot, llmEnriched=true)
       wsBroadcastToViewer(viewerUserId, "prs_update", enrichedSnapshot)
     if failure:
       circuitBreaker.recordFailure()
       [Layer1 결과 유지, 로그만 기록]

5. 비동기 워커 설계
----------------------------------------------------------------

[스케줄링]

workerState: Map<conversationId, {
  timerId,
  isRunning,
  pendingRun,
  lastSnapshotAt (ms),
  lastLlmCallAt (ms),
  latestMessageId
}>

schedule(conversationId, messageId):
  clearTimeout(state.timerId)
  state.latestMessageId = messageId
  state.timerId = setTimeout(() => run(conversationId), 30_000)

[안전장치 체크 (run 진입 시)]

1. 마지막 스냅샷이 30초 미만이면 -> SKIP
2. LLM 마지막 호출이 2분 미만이면 -> Layer1만 실행
3. isRunning = true이면 -> pendingRun = true, SKIP
4. 완료 후 pendingRun = true이면 -> 1회 재실행

[잡 아이덴티티 - 멱등성]

jobId = (conversationId, latestMessageId, modelVersion)
실행 시작 시 latestMessageId 캡처.
실행 완료 후 비교해 무효화된 결과 감지.

[재시도 / 백오프 (Layer 2 LLM 전용)]

시도 1 -> 실패 -> 1000ms + jitter(±300ms)
시도 2 -> 실패 -> 2000ms + jitter(±300ms)
시도 3 -> 실패 -> 포기, Layer1 결과 유지, circuitBreaker.recordFailure()

Layer 1 (pure function)에는 재시도 없음.

[LLM 서킷 브레이커]

상태: CLOSED -> OPEN -> HALF_OPEN -> CLOSED

CLOSED   : 정상. LLM 허용.
OPEN     : 차단. LLM 스킵. Layer1만 사용.
HALF_OPEN: 테스트 1회 허용.

전환:
  CLOSED -> OPEN     : 3회 연속 실패
  OPEN -> HALF_OPEN  : 60초 경과
  HALF_OPEN -> CLOSED: 성공
  HALF_OPEN -> OPEN  : 실패

process-level 싱글톤. 모든 방향 분석이 공유.

[단일 인스턴스 vs 멀티 인스턴스]

현재: 단일 인스턴스 전제 (MVP).
in-memory 디바운스/서킷 브레이커는 단일 인스턴스에서만 안전.
멀티 인스턴스 대응 (MVP 이후): Redis SET NX PX 분산 락 또는 pg-boss 잡 큐.
replit.md에 제한 명시.

6. 방향성 스코어링 설계
----------------------------------------------------------------

[viewer 기반 추출]

featureWindow.myUserId     = viewerUserId   (서버 인증 컨텍스트에서)
featureWindow.partnerUserId = partnerUserId (participants에서 viewer가 아닌 쪽)

messages 분류:
  msg.senderUserId === viewerUserId  -> role: "me"
  msg.senderUserId !== viewerUserId  -> role: "partner"

PRS_AB: viewer=A -> partner=B의 행동 분석 -> "B가 A에게 얼마나 반응적인가"
PRS_BA: viewer=B -> partner=A의 행동 분석 -> "A가 B에게 얼마나 반응적인가"

[참여자 소스 - 현실적 제한]

conversationMembers 테이블 없음.
유일한 소스: distinct senderUserId from chat_messages.

제한 사항 (명시):
1. senderUserId가 NULL인 메시지는 방향성 분석에서 제외
2. 한쪽이 아직 메시지를 안 보냈으면 분석 불가
   -> participants.length < 2이면 SKIP
3. matches 테이블과 연결 없으므로 매치 상태 검증 불가

장기 대응: conversation_members 테이블 도입 또는 matches.conversationId FK 추가.

[WS 뷰어 범위 전달]

broadcastToViewer(conversationId, viewerUserId, payload):
  room = rooms.get(conversationId)
  for client in room:
    if client.userId === viewerUserId AND client.readyState === OPEN:
      client.send(payload)

payload 구조:
{
  type: "prs_update",
  conversationId: "...",
  viewerUserId: 123,
  snapshot: { prsScore, confidenceScore, stage, reasonCodes, ... }
}

7. 캐싱 및 중복 방지 설계
----------------------------------------------------------------

[스냅샷 중복 방지]

새 스냅샷 INSERT 전에 아래 조건을 모두 충족하면 INSERT 생략:
  - prs_score 변화 < 3점
  - stage 동일
  - low_confidence_state 동일
  - reason_codes hash(SHA-256 앞 8자) 동일
  - llm_enriched 동일 (false->true는 INSERT)

최소 하나라도 변화 있으면 INSERT.
latest_interest_snapshots는 중복 여부 무관하게 항상 UPSERT.

임계치 3점 선택 근거:
prsScoring 가중합 특성상 2점 이하 변화는 피처 노이즈 수준.
3점은 UI에서 인지 가능한 최소 변화량.

[Layer 1 -> Layer 2 이중 업데이트 노이즈 방지]

Layer 2 결과는 materialChange 검사 후 WS 전송:
  - |prs_score 차이| >= 5 이거나 llm_enriched false->true인 경우만 전송
  - 기준 미달 시 WS 전송 생략 (DB upsert는 유지)

[증분 피처 추출]

MVP: 매 실행마다 최신 50건 전체 재계산 (< 2ms CPU, < 10ms DB).
허용 범위이므로 증분 캐시 불필요.
interest_feature_cache는 스키마만 설계, MVP에서 미구현.

8. 장애 모드 분석 (A-N)
----------------------------------------------------------------

[A. 방향성 프라이버시 누출]

위험: broadcast(room) 호출 시 PRS_AB가 B에게 전달됨.
영향: B가 A의 PRS를 볼 수 있음. 제품 프라이버시 위반.
완화: broadcastToViewer 구현. client.userId === viewerUserId 조건 필터.
     payload에 viewerUserId 포함해 클라이언트 이중 검증.
잔여 한계: ws.userId가 NULL이면 전달 불가 -> GET 폴백.

[B. 인증 우회 위험]

위험: POST /api/ai/prs에 requireAuth 없음.
     featureWindow.myUserId를 임의로 공급 가능.
영향: 타인 PRS 조작 가능.
완화: interest.router.ts에 requireAuth 추가.
     featureWindow.myUserId를 req.user.userId로 강제 덮어쓰기.
     서버 push에서는 WS의 ws.userId(JWT) 사용.

[C. 참여자 소스 정확성]

위험: distinct sender_user_id 기반 결정. 공식 멤버십 테이블 없음.
영향: 한쪽 미발송 시 참여자 목록 누락. senderUserId=NULL 처리 필요.
완화: participants.length < 2이면 SKIP.
     senderUserId IS NULL 메시지는 "unknown" 처리.
잔여 한계: conversation_members 테이블 없이는 근본 해결 불가.

[D. Score 단위 불일치]

위험: 내부 계산, DB, API, UI 간 단위 불일치.
검증: prsScore 0-100, confidenceScore 0-100 확인됨.
완화: DB REAL 0-100, API 응답 동일. SemanticScores(0-1)는 DB 미저장.

[E. LLM 하드 의존성]

위험: LLM 실패 시 PRS 계산 전체 불가.
완화: Layer 1 (항상 실행) + Layer 2 (선택적).
     LLM 실패해도 Layer1 결과 저장 및 WS 전달.
     llm_enriched=false 플래그로 클라이언트에 명시.

[F. 중복 비동기 실행 / 레이스 컨디션]

위험: 연속 메시지 -> schedule 여러 번 호출 -> 동시 분석 -> DB 중복 INSERT.
완화: isRunning 플래그 + pendingRun 패턴.
     latestMessageId 캡처로 무효화된 결과 감지.
     디바운스 30초로 버스트를 1회로 병합.

[G. 멀티 인스턴스 배포 위험]

위험: in-memory 상태는 단일 인스턴스에서만 안전.
현재: MVP는 단일 인스턴스 전제. replit.md에 명시.
장기: Redis 분산 락 또는 pg-boss 잡 큐.

[H. 스냅샷 폭발 위험]

위험: 히스토리 테이블 무제한 성장.
완화: 중복 방지로 의미 없는 INSERT 차단.
     대화당 viewer당 최대 200건 보존 정책 (MVP 이후 배치 잡).

[I. 전체 재계산 비용]

위험: 매 메시지마다 전체 피처 재계산.
현실: 50건 처리 < 2ms CPU, < 10ms DB. 30초 디바운스로 최악 2회/분.
잔여: 대화 수천 건 성장 시 증분 캐시 도입.

[J. 잘못된 뷰어 전달]

위험: WS prs_update가 잘못된 클라이언트에 전달.
완화: broadcastToViewer의 userId 필터.
     GET API는 req.user.userId와 viewer_user_id 일치 검사.

[K. PII 과잉 저장]

위험: raw 메시지 텍스트를 스냅샷 테이블에 저장.
완화: 저장 없음. reason_codes(정규 코드) + feature_breakdown(집계 숫자)만 저장.

[L. 이중 업데이트 노이즈]

위험: Layer 1 직후 Layer 2가 몇 초 후 다시 WS 전달 -> UI 깜빡임.
완화: materialChange 검사. |prs_score 차이| >= 5 또는 false->true인 경우만 2차 전송.

[M. 메시지 수정/삭제/순서 이슈]

삭제: 다음 schedule 실행 시 자연스럽게 제외.
수정: 수정 기능 없음 (현재 스키마에 updated_at 없음).
순서: ORDER BY created_at ASC로 읽으므로 자연 정렬됨.
잔여: 삭제 후 즉시 재분석 트리거 없음. 다음 메시지 도착 시 재분석.
